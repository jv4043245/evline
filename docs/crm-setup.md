# EVLine CRM setup

Цель CRM: связать рекламу, заявку, заказ, доставку, выручку и маржу в одну цепочку.

## Что уже делает система

1. Формы сайта отправляют заявку в `/api/leads`.
2. Заявка сохраняется в `leads` и автоматически создает запись в `orders`.
3. Каждой заявке, клиенту и заказу присваивается понятный номер: `L-000001`, `C-000001`, `O-000001`. UUID остается внутренним техническим ключом.
4. В заказ попадают контакты, авто, VIN, запчасть/услуга, UTM, `gclid`, `gbraid`, `wbraid`, `fbclid`, landing/page URL, form id/name, submitted time и referrer.
5. Менеджер ведет заказ в `/admin/`: меняет статус, добавляет цену, себестоимость, доставку, трек и заметки.
6. При смене статуса создается событие в истории заказа.
7. Если у клиента привязан Telegram `chat_id`, уведомление отправляется автоматически.
8. Если `chat_id` еще нет, уведомление остается в истории и его можно повторить после привязки.

## Переменные Cloudflare Pages

Обязательные:

- `ADMIN_TOKEN` - резервный токен для админки, если Cloudflare Access еще не включен.
- `TELEGRAM_BOT_TOKEN` - token бота, который отправляет уведомления.
- `CRON_SYNC_TOKEN` - отдельный секрет для фоновой синхронизации треков. Не должен совпадать с `ADMIN_TOKEN`.

Желательные:

- `TELEGRAM_PARTS_CHAT_ID` - внутренний чат заявок по запчастям.
- `TELEGRAM_TECH_CHAT_ID` - внутренний чат заявок по программированию BYD.
- `TELEGRAM_SUPPLIER_CHAT_IDS` - обязательная для китайского направления карта отдельных чатов поставщиков, например `Zeekr=-100111,BYD=-100222` или JSON `{"Zeekr":"-100111","BYD":"-100222"}`.
- `TELEGRAM_CHAT_ID` - общий fallback для обычных менеджерских уведомлений вне китайского направления.
- `TELEGRAM_WEBHOOK_SECRET` - secret token для Telegram webhook.
- `SUPPLIER_TRANSLATION_MODEL` - необязательная модель Cloudflare AI для перевода сообщений поставщика; если не задана, используется модель по умолчанию.

## Telegram webhook

Webhook endpoint:

```text
https://evline.com.ua/api/telegram/webhook
```

Команда установки webhook:

```bash
curl -X POST "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/setWebhook" \
  -H "content-type: application/json" \
  -d '{"url":"https://evline.com.ua/api/telegram/webhook","secret_token":"CHANGE_ME"}'
```

Команда привязки клиента к заказу:

```text
/start order_<order_id>
```

Эта команда показана в карточке заказа. После того как клиент отправит ее боту, CRM запишет `telegram_chat_id` в заказ и карточку клиента.

Если статус поменяли до привязки клиента к боту, уведомление останется в истории со статусом `skipped`. После привязки CRM автоматически попробует отправить последнее неотправленное уведомление. В карточке заказа также есть ручной повтор для неотправленных уведомлений.

Для китайского направления бот работает как мост: поставщик отвечает в своем веб-кабинете, CRM переводит сообщение на русский и отправляет его в персональный чат поставщика из `TELEGRAM_SUPPLIER_CHAT_IDS`. Менеджер отвечает reply на сообщение бота в этом чате, CRM сохраняет русский текст в админке, переводит его на китайский и добавляет в диалог поставщика.

Общего чата для китайского направления нет: если поставщик не указан в `TELEGRAM_SUPPLIER_CHAT_IDS`, Telegram-уведомление по нему не уйдет. Сначала создаем поставщику персональный чат, потом берем его в работу.

Рекомендуемая схема для поставщиков:

- `EVLine Китай Zeekr` - отдельный чат Zeekr.
- `EVLine Китай BYD` - отдельный чат BYD.
- `EVLine Китай Buble` - отдельный чат Buble.

В каждую группу нужно добавить того же Telegram-бота и отправить `/chatid`. Бот вернет Chat ID группы. После этого в Cloudflare Pages нужно прописать:

```text
TELEGRAM_SUPPLIER_CHAT_IDS=Zeekr=-100...zeekr,BYD=-100...byd,Buble=-100...buble
```

Для проверки в админке есть скрытая ссылка на вкладке `Китай`: меню `...` -> `Проверка Telegram`. Она показывает, какие поставщики уже маршрутизируются в отдельный чат, а какие еще не настроены. Полные Chat ID в диагностике маскируются.

## Накат базы

Локальная проверка миграций:

```bash
sqlite3 /tmp/evline-crm.sqlite < migrations/0001_crm.sql
sqlite3 /tmp/evline-crm.sqlite < migrations/0002_orders_tracking.sql
```

Накат на Cloudflare D1:

```bash
npx wrangler d1 migrations apply evline-crm --remote
```

Если Wrangler недоступен, SQL из `migrations/*.sql` можно выполнить в Cloudflare D1 Console. В таком случае команды лучше выполнять отдельными statement'ами, а не одним большим полотном.

После миграции `0007_public_numbers.sql` старые записи получают номера по дате создания:

- `L-000001` - номер лида/первичной заявки;
- `C-000001` - номер клиента;
- `O-000001` - номер заказа.

Дальше номера выдаются автоматически через таблицу `crm_counters`.

## Трекинг доставки

Автоматический трекинг сейчас включен для Meest China:

- карточка заказа хранит последний статус перевозчика, место, дату статуса, время последней проверки и расчетную дату доставки;
- история событий перевозчика пишется в `order_tracking_events`;
- кнопка `Оновити трекінг` в карточке заказа подтягивает свежие события Meest;
- endpoint `/api/admin/tracking/sync` может проверять открытые Meest-заказы пачкой;
- endpoint `/api/cron/tracking-sync` делает то же для фонового расписания и принимает только `CRON_SYNC_TOKEN`;
- первый запуск по уже существующему треку сохраняет базовый статус без клиентского уведомления, следующие изменения создают событие и уведомление в Telegram, если клиент привязан к боту.

Ukr-China пока ведется вручную: их публичный трекинг требует телефон/SMS, поэтому для автоматизации нужен официальный API, webhook или выгрузка от перевозчика.

## Фоновая синхронизация треков

Для автоматической проверки треков раз в несколько часов используется отдельный Cloudflare Worker:

```text
workers/tracking-cron
```

В Pages-проекте `evline` должен быть secret:

```text
CRON_SYNC_TOKEN
```

В Worker `evline-crm-tracking-cron` нужно добавить такой же secret с тем же значением.

Worker вызывает:

```text
https://evline.com.ua/api/cron/tracking-sync
```

Расписание находится в `workers/tracking-cron/wrangler.toml`:

```toml
[triggers]
crons = ["0 */2 * * *"]
```

Это означает запуск каждые 2 часа по UTC. Первый запуск по новому треку фиксирует базовый статус без сообщения клиенту. Следующие изменения статуса отправляются клиенту в Telegram, если заказ уже привязан к боту.

## Статусы заказа

- `new` - заявка получена.
- `accepted` - менеджер принял в работу.
- `proposal_sent` - предложение отправлено клиенту.
- `paid` - заказ оплачен.
- `sourcing_china` - ищем или заказываем в Китае.
- `china_warehouse` - заказ на складе в Китае.
- `left_china` - заказ выехал из Китая.
- `in_ukraine` - заказ в Украине.
- `ready_for_pickup` - готово к выдаче/отправке.
- `completed` - завершено.
- `canceled` - отменено.

## Что менеджер обязан заполнить для аналитики

Минимум для каждого заказа:

- телефон клиента;
- авто/модель;
- VIN, если есть;
- запчасть или услугу;
- статус заказа;
- выручку;
- закупку;
- доставку;
- мито/оформление, обработку и другие расходы, если они есть;
- рекламную стоимость заказа, если известна;
- причину отмены, если заказ отменен.

Для логистики:

- перевозчик;
- трек-номер;
- ссылка на трек;
- статус доставки.

## Как считать эффективность

CRM считает:

- заявки;
- активные и завершенные заказы;
- выручку;
- валовую маржу;
- CPL;
- CPA;
- ROAS;
- profit ROAS;
- эффективность источников и кампаний.

Чтобы эти цифры были честными, в блоке “Витрати на рекламу” нужно регулярно вносить расходы Google Ads/Facebook по датам и кампаниям. Для Google Ads лучше использовать `utm_campaign={campaignid}` в Final URL suffix: тогда расходы из Google Ads cost sync совпадают с заказами по campaign id.

## Google Ads offline conversions

CRM готовит события:

- `lead` - заказ создан из заявки;
- `paid` - заказ дошел до оплаты или более позднего статуса.

По умолчанию в Google Ads отправляется value по марже, а не по выручке. Если нужна выручка, задайте `GOOGLE_ADS_CONVERSION_VALUE_MODE=revenue`.

Перед отправкой оплаченного заказа менеджер должен заполнить финансы заказа. Если финансы изменились до upload, CRM пересчитает queued/failed conversion rows. Если conversion уже отправлена, ее значение не меняется автоматически.

## Отчет по ключевым словам

В разделе “Аналітика і реклама” есть блок “Прибутковість ключів”. Он нужен для практических решений по рекламе:

- какие ключи масштабировать;
- какие поисковые запросы проверить вручную;
- какие запросы становятся кандидатами в минус-слова;
- где клики есть, но маржи нет.

Основа отчета:

- расходы, клики и показы из Google Ads keyword/search-term sync;
- лиды и заказы из CRM по `utm_campaign + utm_term`;
- маржа из заказа;
- Google-side conversion value по search terms после загрузки offline conversions.

Отчет становится точнее, когда менеджеры регулярно закрывают финансы заказа, а Google Ads cost sync запускается ежедневно.
